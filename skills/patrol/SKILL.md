# patrol

整理の第二の入口。外を巡回して「考えた方が良いこと」を拾い、spar（鍛える）または直接アクションに渡す。spar が頭の中の悩みを内から引き出すのに対し、patrol は外部（GitHub）に散らばった論点を外から持ち込む。情報の向きが逆で、悩みがまだ言語化されていない／やること自体を探している状態の入口になる。

**patrol は結論を出さない。** 「これは作業だ」と決め切るのは下流（spar / アクション）の仕事。patrol は「ここ、考えた方がよくない？」と論点を置くだけ。

## 1. 巡回対象の決定

引数（`<org>` または `<owner>/<repo>` 群）で受け取る。**config ファイルは作らない・要求しない**（organize は作業ファイルを作らない。空でも即使える）。

- 引数があればそれを対象にする。
- 引数が無ければ、今いるリポの owner / org を推定して候補に挙げ、一問だけ「ここを巡回でよいか？」と確認する。

`gh repo list <org>` で対象リポを列挙する。

## 2. 二段構えの走査

org 全体を毎回精読するとコストで破綻する。広く浅く絞ってから、狭く深く読む。

- **Pass 1（広く・浅く・安価）**: 各リポをメタデータ級の signal だけで横断ランク付けする。深い精読はしない。
  - 最終 push の古さ / commit 数・頻度（＝活動量）
  - open issue・PR の滞留（滞留日数は `gh pr list` / API で安価に取れる）
  - README の有無、説明文の有無
  - → 「**土台の弱さの匂い × 活動量**」でリポを並べる
- **Pass 2（狭く・深く）**: 上位数件だけ、commit 履歴・ファイル構成・既存 doc（`docs/adr/` 含む）を精読し、**具体的な「考えた方が良いこと」**を言語化する。原因となる doc に逆引きする（→ 3）。

## 3. 診断レンズ（判定ロジック）

場当たりなヒューリスティクスは使わない。判定は organize の核心 2 つから演繹する:

| organize の核心 | well-organized な状態 | patrol が拾う負債 |
|---|---|---|
| **ちゃんと言語化** | 目的・用語・構造・決定が複雑さに見合う程度に言語化されている（＝ Catalog の各 doc） | **言語化負債**: churn / revert＝決定が未言語化（→ ADR）／ commit での用語揺れ（→ CONTEXT）／スコープ混乱（→ BRIEF）／構造が辿れない（→ ARCHITECTURE）／README 不在 |
| **GitHub を使った流れで整理** | 作業が issue → branch → PR → merge の規律ある履歴を通る | **flow 負債**: main 直 commit ／作業に対応する issue が無い ／**PR 放置** ／ branch 運用の崩れ |

判定を一文に畳む:

> patrol は、リポの「**複雑さ × 活動量**」に対して「**言語化の度合い + GitHub flow の規律**」がどれだけ追いついていないか（負債）を測り、ギャップの大きい順に「考えた方が良いこと」として置く。

**弁別子（lazy 原則を守る最重要ルール）**:

- **「doc が無い」では flag しない。** doc 不在そのものは負債ではない（organize の doc は lazy 作成＝ signal が出るまで作らないのが正常）。
- **負債は活動量でスケールする。** 活動に対して言語化／flow が負けているときだけ負債。commit 数個・放置気味の若いリポは、doc が無くてもギャップが小さく、自然に足切りされる。
- 言い換えると patrol が検出するのは「signal が repo の歴史で繰り返し surface しているのに、誰も doc を作っていない／flow に乗せていない」状態。「signal が出たら作る」原則の自然な拡張であり、督促チェッカーではない。

調べ方の規律: 巡回は read-only（`gh` / API）。外部検索に機密を流さない。

## 4. 出力と下流への振り分け

- 出力は**揮発するランク付きリスト**（会話に出すだけ。commit しない・ファイル化しない。organize の「作業ファイルを作らない」規律に沿う）。各項目に負債の種類と推奨下流を添える。
- **人間が 1 件選ぶ。** 何を整理するかの選択権は人間が握る（自動で 1 件に決め打って丸投げにしない）。patrol は複数を一度に surface するが、深掘りは 1 件ずつ直列に回す。
- 選ばれた 1 件を下流に渡す:

| 負債の性質 | 下流 |
|---|---|
| 言語化負債／未解決の論点で止まっている | `spar`（鍛える）→ 出口（残す） |
| flow 負債で打ち手が明確（merge 待ちの放置 PR・close すべき PR・issue 化すれば済む作業） | spar を経ず直接アクション（`create-pr` の続行 / `create-issue` / close） |

patrol は下流を**提案**するが、spar に通すかアクションに回すかを選ぶのは人間。

## 5. 締め

巡回は直列に回す。1 件を下流に渡して片付いたら、残りのリストに戻って次を選べるよう促す。どの項目を扱っても、最後に「次に何をするか」を一言で示して終える（伴走）。
