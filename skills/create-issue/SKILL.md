# create-issue

GitHub issue を、このファイル末尾の「issue テンプレート」に沿って起票する。

## 手順

1. **材料集め**: 会話・リポジトリ・関連 doc（BRIEF.md / ARCHITECTURE.md 等）から、テンプレートを埋めるのに必要な情報を集める。コードやリポジトリを見れば分かることは利用者に聞かない。
   - 材料が薄く、テンプレートを質高く埋めきれない（背景・スコープ・成功基準が立たない、優先度・難易度の推奨値に根拠を添えられない等、計画自体が定まっていない）と気づいたら、起票前に「これは spar で計画を鍛えてからの方が良い issue になりそうです」と一度だけ提案し、利用者の判断を仰ぐ。ただし調べれば埋まる空欄は対象にしない。
2. **下書き**: 末尾のテンプレートの全セクションを埋めた本文を下書きする。
   - 下書きファイルは作らず、本文は会話で提示する
   - 該当しないセクションは `N/A`（一言理由つき）とし、セクション自体は削らない
   - 埋められない重要な空欄だけ、一問一答・推奨案付きで利用者に確認する。優先度・難易度は必ず推奨値と根拠を添える
   - 工数見積は確実性レベルを選び、バッファ込みの最終見積まで出す
3. **タイトル**: Conventional Commits の型に合わせる（例: `feat: ○○を追加する`）
4. **起票**: 下書きを利用者に提示して確認を得てから起票する。本文は stdin で渡す:
   `gh issue create --title "<title>" --body-file -`
   - 起票が成功したら、続けて `gh issue view <#> --web` で既定ブラウザに該当 issue を開き、目視確認できるようにする。失敗時は開かない
   - 既存 issue を拡張・修正する場合は `gh issue edit` を使う
5. **次へ**: 起票後、すぐ着手するか問う。着手するなら `feature/<#>-<summary>` ブランチを切って作業フェーズへ進む。

## 禁止

- issue 本文に AI 生成の旨を記載しない（`🤖 Generated with...` 等を付けない）
- 機密情報（トークン・内部 URL 等）を本文に含めない
- テンプレートの構造を勝手に変えない。不足セクションが必要なら利用者に確認してから足す

## issue テンプレート

以下を単一の真実として埋める。編集はこの区画だけで完結する。

````markdown
## 📋 概要

[1-2文で変更内容を簡潔に説明]

関連 issue: [関連 Issue があれば #番号]

---

## 🎯 背景・課題

### 現状の問題
- [現在どのような問題があるか]
- [なぜこの変更が必要なのか]

### 課題
- [具体的な課題1]
- [具体的な課題2]

---

## 🎯 目的・ゴール

### 主目的
[この変更で達成したいこと]

### 副次的目標
- [追加で達成したいこと1]
- [追加で達成したいこと2]

---

## 📖 要件定義

### 機能要件 (FR)

#### FR-1: [機能名1]
- [詳細な要件説明]

#### FR-2: [機能名2]
- [詳細な要件説明]

### 非機能要件 (NFR)

#### NFR-1: パフォーマンス
- [要件]

#### NFR-2: セキュリティ
- [要件]

#### NFR-3: 保守性
- [要件]

---

## 🛠️ 技術仕様

### アーキテクチャ
[構成図、フロー図など]

### 実装詳細
[具体的な実装方法、コード例など]

---

## ✅ 受け入れ基準

### Must Have
- [ ] [必須要件1]
- [ ] [必須要件2]

### Should Have
- [ ] [推奨要件1]

### Nice to Have
- [ ] [オプション要件1]

---

## ⏱️ 工数見積

### タスク分解

| タスク | 内容 | 見積工数 |
|--------|------|----------|
| 設計 | [設計内容] | X.Xh |
| 実装 | [実装内容] | X.Xh |
| テスト | [テスト内容] | X.Xh |
| ドキュメント更新 | [更新内容] | X.Xh |

合計見積: X時間 (X営業日)

### 確実性レベル

| レベル | 説明 | バッファ |
|--------|------|---------|
| 高 | 過去に類似実装あり、よく知っている技術 | +10% |
| 中 | 一般的な実装パターン、ある程度経験あり | +25% |
| 低 | 初めての技術、複雑な仕様 | +50% |

最終見積: X時間 (バッファ込み)

---

## 🧪 テスト計画

### TC-1: [テストケース名]
- 手順:
  1. [手順1]
  2. [手順2]
- 期待結果: [期待される結果]

---

## ✅ Definition of Done

- [ ] 受け入れ基準 Must Have 全て満たす
- [ ] テストケース全て合格
- [ ] ドキュメント更新完了
- [ ] コードレビュー承認
- [ ] 本番環境デプロイ・検証完了 (該当する場合)

---

## 📚 関連資料

- [関連ドキュメント、外部リンク]

---

**優先度**: High / Medium / Low
**難易度**: High / Medium / Low
**依存関係**: [依存する Issue や前提条件]
````
