# spar

整理の入口で、計画を鍛えるための壁打ちをする。一問一答で矛盾・曖昧さ・未検証の前提を突き、対話で surface した signal から必要な doc を逆引きし、最後に出口へ案内する。organize の入口 skill。

## 1. 現状把握の対話

対話に入る前に、案件の既存 doc（`docs/adr/` を含む）を読み、照合の土台にする。

進め方:

- **一問一答**: 一度に 1 つだけ聞き、回答を待ってから次へ進む。複数の質問を並べない
- **推奨案付き**: 各問いには必ず推奨案と根拠を添える
- **決定木を一枝ずつ**: いちばん太い分岐（後続の問いの前提になる分岐）から順に潰す
- **枝を潰し切るまで止めない**: 未解決の太い分岐が残るうちは出口に進まない。一段落の判断は「太い分岐がすべて潰れたか」で行う（signal が出揃った・doc が埋まった、ではない）

鍛え方（壁打ちの核）。質問は集めるためでなく、計画を鍛えるためにある:

- **共有理解に達するまで問い尽くす**: 表面的な合意で切り上げない。計画のあらゆる面を、利用者と同じ絵が描けるまで掘る
- **既存 doc との矛盾を即指摘**: 案件に存在するすべての doc が照合対象。食い違う発言が出たら、その場で突く
  - 用語: 「CONTEXT.md では X は◯◯だが、いま △△ の意味で使っていないか？」
  - スコープ: 「BRIEF.md は □□ を含まないとしているが、いまの話は踏み込んでいないか？」
  - 構造: 「ARCHITECTURE.md は同期 API を持たない構成としているが、いまの案はそれを破っていないか？」
  - 過去の決定: 「ADR-0002 は Postgres と決めているが、覆す判断か？」
- **曖昧語の正準化**: 曖昧・多義的な語が出たら、正準の用語を提案して一つに決める（「『アカウント』は Customer と User のどちらか？」）
- **具体シナリオでストレステスト**: 概念の関係が語られたら、境界を突くエッジケースのシナリオを発明してぶつけ、曖昧さを残さない
- **コードとの矛盾を突きつける**（コード有り案件のみ。コード無し案件〔マーケ・リサーチ等〕では skip）: 利用者の説明とコードの実態が食い違ったら surface する（「コードは Order 全体を cancel しているが、いま部分キャンセル可能と言った。どちらが正しい？」）

対話で確定した内容（解消した矛盾・確定した用語）も signal として扱う（→ 2. signal の逆引き）。

調べ方:

- **調べて分かることは聞かない**: コード・リポジトリ・web で答えが出る問いは自分で調べる
- 「事実を確認しないと答えが出ない」論点が出たら、その場で裏取りする
- 重い調査（多源・要検証・レポート級）は `deep-research` skill に委譲する
- 検索クエリ等で外部に送る内容に、対話で出た機密情報を含めない
- 調べた問いと結論（出典つき）も signal として扱う（→ 2. signal の逆引き）

## 2. signal の逆引き（Doc Catalog）

対話に入る前に [catalog.md](./catalog.md) の signal 列に目を通しておく。対話中に話題がそのいずれかに踏み込んだら（= signal が surface したら）、catalog を逆引きして該当 doc に当てる。doc ごとに要否を聞いて回る走査はしない。

- 該当 doc が**まだ無ければ、作成を能動提案する**（lazy 作成。signal が出るまで作らない。空の placeholder は一切置かない）
- 該当 doc が**既に有れば、その場で inline 更新する**（まとめて後でやらない）
- Catalog にない doc が必要になったら、必要性を判断したうえで catalog.md に行を足す

## 3. 出口の分岐

太い分岐がすべて潰れたら（→ 1. の「枝を潰し切るまで止めない」）、同じ作法（一問一答・推奨付き）で出口を 1 問だけ問う:

| 出口 | いつ | 次の誘導 |
|---|---|---|
| 残す | 残す価値がある、または着手する | spar で作成 / 更新した md を添えて `create-issue` skill へ（issue から main 反映を開始する）。md が無い純タスクなら issue だけ。新規案件はまず `git init` / `gh repo create` |
| 何も残さず動く / 終える | 整理で十分、git 管理不要 | そのまま実行 / 終了 |
| handoff して中断 | 次のセッション、または別の AI / ツールに引き継ぎたい | `handoff.md`（commit しない）に到達点・未解決論点・次の一歩を書いて終了。別 AI / ツールへのバトンタッチも想定し、handoff は必ず 1 ファイルにする（ディレクトリ化・複数ファイルへの分割をしない） |

### 「残す」を選んだら、続けて一問

「PRまで自動実行しますか？」を推奨付きで問う（offered affordance。強制しない）。これは organize の「自動実行モード」の唯一の起動点であり、その運用契約の正準定義でもある:

- **no（既定）**: 従来どおり `create-issue` から逐次に進み、各 skill の事前確認ゲートで都度確認する。
- **yes**: フォアグラウンドのまま `create-issue` → `commit` → `create-pr` を連鎖実行する。このモードの契約:
  - **省くのは事前確認ゲート 3 つだけ** — create-issue の起票前確認 / commit のメッセージ確認 / create-pr の作成前確認。下書きを提示して承認を待たず、そのまま実行する。
  - **終点は PR 作成＋ブラウザ表示**（create-pr が PR を `--web` で開く）。**merge は畳まない**（squash merge は人間）。
  - 事後レビューは成果物で行う — create-issue が issue を、create-pr が PR をブラウザで開くので、人間はそこで検証し、必要なら `gh issue edit` / PR 編集で直す。
  - **発火は素性で固定**: 自動実行モードは「この出口で yes を選んだ連鎖」に限る。各 skill を単発で呼んだときは事前確認を省かない（fail-closed。迷ったら確認する）。
  - subagent / worktree は使わない（フォアグラウンド）。真の探索（成果物が返らないタスク）はこのモードに乗せない。

## 4. 締め

どの出口を通っても、最後に「次に何をするか」を一言で示して終える（伴走）。
