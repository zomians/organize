# pwd

整理の現在地を判定し、必要なら現状把握の対話で頭を整理し、出口へ案内する。organize の入口 skill。

## 1. 現在地判定

ディレクトリの状態・git/issue の状態・会話の意図を観察し、現在地を判定する。観察で分かることは聞かない。判定が曖昧なときは決めつけず、推奨案付きで **1 問だけ**聞く。

| 現在地 | 兆候 | 案内先 |
|---|---|---|
| 空ディレクトリ / 新ネタ | ファイルがほぼ無い・初めての話題 | 「2. 現状把握の対話」へ |
| doc 整備途中 | 大文字 md が一部あり、会話がその続き | 対話の続行（未解決の論点から再開） |
| 整理は済み、残すか判断する局面 | 頭は整理済み、成果の扱いだけ未定 | 「4. 出口の分岐」へ |
| 既発行 issue に対応 | open issue があり着手の意図 | 対処フェーズ（`feature/<#>-<summary>` branch → 作業 → PR） |
| 作業途中（branch 上） | feature branch 上に未コミット / 未 PR の変更 | commit / PR 完了へ |

git/issue の状態は `git status` `git branch --show-current` `gh issue list` 等で観察する（リポが無ければ skip）。

## 2. 現状把握の対話

作法:

- **一問一答**: 一度に 1 つだけ聞き、回答を待ってから次へ進む。複数の質問を並べない
- **推奨案付き**: 各問いには必ず推奨案と根拠を添える
- **決定木を一枝ずつ**: いちばん太い分岐（後続の問いの前提になる分岐）から順に潰す
- **調べて分かることは聞かない**: コード・リポジトリ・web で答えが出る問いは自分で調べる

鍛え方（壁打ちの核）。質問は集めるためでなく、計画を鍛えるためにある。対話に入る前に案件の既存大文字 md（と `docs/adr/`）を読み、照合の土台にする:

- **共有理解に達するまで問い尽くす**: 表面的な合意で切り上げない。計画のあらゆる面を、利用者と同じ絵が描けるまで掘る
- **既存 doc との矛盾を即指摘**: 案件に存在するすべての doc が照合対象。食い違う発言が出たら、その場で突く
  - 用語: 「CONTEXT.md では X は◯◯だが、いま △△ の意味で使っていないか？」
  - スコープ: 「BRIEF.md は □□ を含まないとしているが、いまの話は踏み込んでいないか？」
  - 構造: 「ARCHITECTURE.md は中央オーケストレータを作らないとしているが、それは矛盾しないか？」
  - 過去の決定: 「ADR-0002 は Postgres と決めているが、覆す判断か？」
  - 矛盾が解消したら、該当 doc をその場で inline 更新する（→ 3. signal の逆引き）
- **曖昧語の正準化**: 曖昧・多義的な語が出たら、正準の用語を提案して一つに決める（「『アカウント』は Customer と User のどちらか？」）。確定した用語はその場で CONTEXT.md へ
- **具体シナリオでストレステスト**: 概念の関係が語られたら、境界を突くエッジケースのシナリオを発明してぶつけ、曖昧さを残さない
- **コードとの矛盾を突きつける**: 利用者の説明とコードの実態が食い違ったら surface する（「コードは Order 全体を cancel しているが、いま部分キャンセル可能と言った。どちらが正しい？」）

リサーチ（内蔵）:

- 「事実を確認しないと答えが出ない」論点が出たら、その場で裏取りする
- 重い調査（多源・要検証・レポート級）は `deep-research` skill に委譲する
- 検索クエリ等で外部に送る内容に、対話で出た機密情報を含めない

コードとの照合はコード有り案件のみ。コード無し案件（マーケ・リサーチ等）では skip する。

## 3. signal の逆引き（Doc Catalog）

対話中に「特定の doc が必要だ」と示す signal が surface したら、[catalog.md](./catalog.md) を逆引きし、該当 doc の作成 / 更新を能動提案する。

- 全 doc **lazy 作成**。signal が出るまで作らない。空の placeholder は一切置かない
- **inline 更新**: 関連話題が出たその場で更新する（まとめて後でやらない）
- **大文字 md = 残す（git 管理対象）/ 小文字 md = 残さない（揮発）**の命名規約に従う
- Catalog にない doc が必要になったら、必要性を判断したうえで catalog.md に行を足す

## 4. 出口の分岐

対話が一段落したら、同じ作法（一問一答・推奨付き）で出口を 1 問だけ問う:

| 出口 | いつ | 次の誘導 |
|---|---|---|
| A. doc に残す（大文字 md） | 残す価値がある | Catalog に従い doc を作成 / 更新 → commit へ |
| B. issue を直接発行 | やることが明確で、doc を経ずタスク化したい | `create-issue` skill へ |
| C. doc も issue も両方 | 残す価値があり、かつ着手もする | doc 化 → `create-issue` skill へ |
| D. 何も残さず動く / 終える | 整理で十分、git 管理不要 | そのまま実行 / 終了 |
| E. handoff して中断 | 整理が途中だがセッションが尽きた | 小文字 `handoff.md` に到達点・未解決論点・次の一歩を書いて終了 |

doc 化は必須フェーズではなく、A / C を選んだときだけ通る分岐である。

## 5. 締め

どの出口を通っても、最後に「次に何をするか」を一言で示して終える（伴走）。
