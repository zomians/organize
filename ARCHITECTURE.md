# ARCHITECTURE.md

zomians/organize plugin の構造・skill の座組み・運用規約。なぜその形かを含む。

## 全体像

organize は Claude Code plugin。空ディレクトリでも skill が即使えるよう全ディレクトリで有効。改善は plugin update で全案件に一括反映される。

整理のワークフローを直線の手続きではなく、**現状把握をハブにした放射状**として捉える。pwd で頭が整理できたら、その成果を「どう残すか（あるいは残さないか）」が出口として分岐する。

## skill の座組み

フェーズ間の遷移を中央のオーケストレータが統括するのではなく、各 skill が末尾で次を能動的に促し、現在地を判定する軽い入口（pwd）を置く。重い中央制御は作らない（skill が重くなり、途中フェーズだけ使う自由度が下がり、層が増えると分類論が再燃するため。Claude Code の skill は本来 signal 駆動で疎結合に発火する）。

### pwd（入口兼・現状把握）

現在地判定と現状把握の対話を兼ねる単一のエントリ skill。

**現在地判定**: ディレクトリの状態・git/issue の状態・会話の意図から、利用者が今どのフェーズにいるかを判定し、適切なフェーズへ案内する。判定が曖昧なときは決めつけず 1 問だけ聞く。想定する現在地:

- 空ディレクトリ / 新ネタ → 現状把握から
- doc 整備途中 → 対話の続行
- 整理は済み、残すか判断する局面 → 出口の分岐へ
- 既発行 issue に対応 → 対処フェーズ（branch → 作業 → PR）へ
- 作業途中（branch 上）→ commit / PR 完了へ

**対話の作法**: 一問一答・推奨案付き・決定木を一枝ずつ潰す。各質問への回答を待ってから次へ進む。コードや web で答えが出る問いは、聞かずに自分で調べる。

**リサーチ内蔵**: 「事実を確認しないと答えが出ない」論点が出たら、その場で裏取りする。重い調査は `deep-research` skill に委譲する。リサーチを独立 skill にはしない。

**コードとの照合はコード有り案件のみ**。コード無し案件（マーケ・リサーチ等）では skip する。

### 後続フェーズ skill（未設計）

doc 化 / repo 化 / issue 作成 / commit / PR 完了。pwd の設計確定後に書き下ろす。それぞれ末尾で次フェーズを促す。

## Doc Catalog（plugin 同梱の辞書）

pwd は Catalog を辞書として持ち、対話中に signal が surface したら逆引きして該当 doc の作成/更新を能動提案する。Catalog は案件側の CLAUDE.md ではなく plugin に同梱する（plugin update で全案件に一括反映でき、案件ごとに Catalog がドリフトしない）。skill 側に doc リストを埋め込まず、Catalog を単一の真実とする。

### doc 作成のルール（全 doc 共通）

- **全 doc は lazy 作成**。signal が出るまで作らない。空の placeholder ファイルは一切置かない。doc の違いは「Catalog 辞書に載っているか否か」だけで、doc 別に常設/lazy を決めることはしない
- **inline 更新**: 関連話題が出るたびその場で更新する（まとめて後でやらない）
- **配置**: 大文字 + `.md` でリポ root に置く。肥大化したら `docs/` 配下を検討
- **Catalog にない doc** が必要になったら、その場で必要性を判断し、表に行を足す

### 大文字 / 小文字の規約（残す / 残さないの分岐）

情報を「残すべきか否か」は命名で表現する:

- **大文字 md（BRIEF.md / AUDIENCE.md 等）= 残す = git 管理対象**
- **小文字 md（handoff.md 等）= 残さない = その場限り / 揮発**

この分岐は仕組み（gitignore 等）で強制せず、規約として運用する。commit / repo 化 skill が大文字 md を残す対象として扱う。

### Catalog

すべて領域非依存。signal が surface したら pwd が逆引きして提案する。1 行ガードは「その doc に何を書き、何を書かないか」。ガードが未記載の doc は、必要になったときに詰める。

| ファイル名 | 主問い | signal シナリオ / 1 行ガード |
|-----|-------|----------------|
| `BRIEF.md` | 何を・なぜ・どうなれば成功か | 目的・スコープ・成功基準のみ。決定の経緯は ARCHITECTURE/ADR、実装タスクは issue へ逃がす |
| `AUDIENCE.md` | 誰のため・読み手の前提と期待は | 読み手像のみ。読み手が自分自身でも明記（いつの自分か・何を忘れているか）。施策・コピーは別 doc |
| `CONTEXT.md` | この案件で使う言葉は何を指すか | 固有用語の正準定義のみ。実装詳細・spec・scratch・一般概念は載せない |
| `ARCHITECTURE.md` | 全体の構造・座組みは。なぜその形か | 構造と関係・選定理由のみ。spec・実装計画は書かない。尖った個別決定は ADR へ |
| `docs/adr/` | なぜこの個別決定をしたか | ARCHITECTURE が描く構造の中で、後戻り困難 × 文脈なしには驚く × 実トレードオフ の 3 条件を満たす決定のみ。1〜3 文で文脈/決定/理由。ADR は architecture ありき。3 条件を満たさないなら作らない（sparingly） |
| `RESEARCH.md` | 何を調べ、何が分かったか | 調べた問いと結論（出典つき）のみ。生ログ・検索過程・未消化メモは残さない（それは小文字の使い捨て md へ） |
| `README.md` | この案件は何で、どう使うか | 概要と使い方のみ。設計詳細は各成果物 doc へ |
| `STAKEHOLDERS.md` | 誰が関与・承認・意思決定するか | 依頼主・承認者・関係チームの役割を整理しているとき |
| `API.md` | 何を公開するか（endpoint・契約） | API endpoint・OpenAPI・公開契約の議論 |
| `DATA.md` | 何のデータを扱うか（スキーマ・ETL） | データソース・スキーマ・ETL・集計の議論 |
| `CONTENT.md` | 何のコンテンツを作るか | 記事・素材・編集ワークフローの議論 |
| `DESIGN.md` | 何を見せるか（UI 構造・visual taste） | UI・コンポーネント・レイアウト・visual taste の議論 |
| `BRAND.md` | なぜこのトーンか | ブランド・トーン・コピーライティング方針の議論 |
| `LEGAL.md` | なぜこの制約があるか | 契約・ライセンス・プライバシー・コンプライアンスの議論 |
| `DEPLOYMENT.md` | いつどう出すか | デプロイ・リリース・環境変数・インフラの議論 |
| `SEO.md` | どこで見つけてもらうか | 検索最適化方針・サイト構造の議論 |
| `SECURITY.md` | どう守るか | 認証・機密・脆弱性・コンプライアンスの議論 |
| `ANALYTICS.md` | どう測るか | KPI・計測設計・イベント命名・ダッシュボードの議論 |
| `SALES.md` | どう売るか | 提案書・価格・営業フローの議論 |
| `FINANCE.md` | いくらで請け、いくら残るか | 見積・請求・入金・原価・粗利の議論 |
| `ACCOUNTING.md` | お金の動きをどう記録するか | 仕訳・勘定科目・税区分の議論 |

`CONTRIBUTING.md` は案件側には基本作らない。規約（Conventional Commits / squash merge / branch 削除 / AI 生成の旨を記載しない 等）は plugin が全案件に一括で持たせる。案件固有の規約が出たときだけ lazy 作成する。

## pwd の出口

pwd は対話が一段落したら、同じ作法（一問一答・推奨付き）で出口を 1 問問い、選ばれた先へ案内する。出口は 5 通り:

| 出口 | いつ |
|---|---|
| A. doc に残す（大文字 md） | 残す価値がある |
| B. issue を直接発行 | やることが明確、doc を経ずタスク化したい |
| C. doc も issue も両方 | 残す価値があり、かつ着手もする |
| D. 何も残さず動く / 終える | 整理で十分、git 管理不要 |
| E. handoff して中断（小文字 handoff.md・揮発） | 整理が途中だがセッションが尽きた。続きを次へ渡す |

doc 化は必須フェーズではなく、出口 A/C を選んだときだけ通る分岐である。

## 規律

plugin が全案件に持たせる規律:

- **DRY / YAGNI / TDD / Frequent commits**
- **Conventional Commits**。1 PR = 1 squashed commit に収まる粒度
- GitHub Flow / branch 命名 `feature/<#>-<summary>` / squash merge → branch 削除
- **コミットメッセージ・PR 本文・Issue 本文に AI 生成の旨を記載しない**（`🤖 Generated with...`、`Co-Authored-By: Claude...` 等は付与しない）
